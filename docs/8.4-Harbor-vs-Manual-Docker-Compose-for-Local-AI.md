# Harbor vs Manual Docker Compose for Local AI

A manual Docker Compose local AI setup can work well when you only need one or two containers. Harbor is useful when the setup becomes a self-hosted AI stack: a frontend, one or more model backends, search, speech, image generation, proxies, coding tools, and shared configuration.

This guide explains what Harbor adds beyond writing raw Compose files yourself, and when a plain Docker Compose local AI project is still enough.

## The Manual Compose Path

With manual Compose, you own every part of the stack:

- Choosing each service image and version
- Writing service names, ports, volumes, networks, and health checks
- Connecting frontends such as [Open WebUI](./2.1.1-Frontend&colon-Open-WebUI.md) to backends such as [Ollama](./2.2.1-Backend&colon-Ollama.md), [llama.cpp](./2.2.2-Backend&colon-llama.cpp.md), or [vLLM](./2.2.3-Backend&colon-vLLM.md)
- Adding optional services such as [SearXNG](./2.3.1-Satellite&colon-SearXNG.md), [Speaches](./2.2.14-Backend&colon-Speaches.md), [ComfyUI](./2.1.2-Frontend&colon-ComfyUI.md), [LiteLLM](./2.3.5-Satellite&colon-LiteLLM.md), or workflow tools
- Keeping environment variables and service-specific config files synchronized
- Remembering which compose files, model settings, and ports belong to each workflow

That control is valuable. It is also the maintenance cost Harbor tries to reduce for common local AI stack patterns.

## Harbor Still Uses Docker Compose

Harbor is not a replacement runtime for Docker Compose. It builds on Compose and keeps the generated command inspectable:

```bash
harbor cmd -h
```

That command shows the base `compose.yml`, selected service files, and any matching integration files. The Compose setup guide documents the matching rules: direct service files, multi-service files, cross-service files, environment-specific files, default services, and shared environment handling. See [Harbor Compose Setup](./6.-Harbor-Compose-Setup.md) for the details.

The difference is that Harbor chooses the relevant files from service handles instead of requiring you to maintain a separate compose command for every combination.

## Service Selection by Handles

Manual Compose selection usually means editing YAML, maintaining several override files, or keeping long shell commands around. Harbor uses service handles:

```bash
harbor up
harbor up searxng
harbor up webui ollama searxng llamacpp
```

The default `harbor up` starts the configured default services, which are [Ollama](./2.2.1-Backend&colon-Ollama.md) and [Open WebUI](./2.1.1-Frontend&colon-Open-WebUI.md) in the default profile. Additional handles add services to that selected stack. You can also manage defaults directly:

```bash
harbor defaults
harbor defaults add searxng
harbor defaults rm ollama
```

This service selection model is the main day-to-day advantage for a Docker Compose local AI setup: the stack changes through service names, not by editing YAML each time.

## Cross-Service Integrations

Many local AI tools are useful alone, but better when another service is present. Harbor models that with cross-service integration files such as `services/compose.x.<service>.<integration>.yml`. Those files are included only when all required handles are selected.

For example, adding SearXNG can make web search available to compatible services, and adding speech or image services can mount extra configuration into frontends that know how to use them. The exact behavior depends on the service, but the pattern is consistent: the base service stays usable alone, and the integration file applies when the related services run together.

With manual Compose, you can build the same behavior yourself. Harbor keeps those common combinations in the project so you can start with the service handle and inspect the resulting Compose selection afterward.

## Shared Configuration and Service Overrides

Manual Compose local AI stacks often accumulate scattered `.env` files, copied examples, and per-service config fragments. Harbor centralizes common settings in its profile config and exposes them through the CLI:

```bash
harbor config get webui.host.port
harbor config set webui.host.port 8080
harbor env ollama OLLAMA_DEBUG 1
```

`harbor config` works with Harbor's shared `.env` profile, while `harbor env <service>` writes service-specific override variables. Many service CLIs are aliases over config keys, so common model settings can be changed without remembering the underlying environment variable:

```bash
harbor vllm model Qwen/Qwen3.5-4B
harbor llamacpp args "-c 4096"
```

This shared configuration is convenient, but it is also intentionally transparent: the files live in the Harbor workspace and can be inspected when needed. See the [Harbor User Guide](./1.-Harbor-User-Guide.md#configuring-services) and [CLI Reference](./3.-Harbor-CLI-Reference.md#setup-management-commands) for the supported commands.

## CLI and App Workflows

Raw Compose gives you `docker compose up`, `down`, `logs`, `exec`, and related primitives. Harbor wraps those primitives with local AI-specific workflows:

```bash
harbor up
harbor open
harbor ps
harbor pull llama3.2
harbor shell llamacpp
harbor exec webui ls /app/backend/data
```

The [Harbor CLI Reference](./3.-Harbor-CLI-Reference.md) documents the command surface. The [Harbor App](./1.1-Harbor-App.md) provides a GUI for managing the same kind of setup when you do not want to work only from the shell.

For a self-hosted AI stack, the value is not that these commands are impossible with Compose. The value is that they use the same Harbor service handles, profiles, ports, and integrations as the rest of the stack.

## Harbor Launch for Local Tools

Manual Compose can expose an OpenAI-compatible backend, but each host tool still needs its own provider configuration. Harbor Launch connects supported installed coding and agent CLIs to a Harbor backend from the current project directory:

```bash
harbor launch --backend ollama --model qwen3.5:4b codex
harbor launch --backend llamacpp --model Qwen3.5-4B opencode
harbor launch --config opencode
```

Launch can select a reachable backend, discover models from compatible `/v1/models` responses, pass through tool arguments, and write config for supported tools. It is especially useful when the same local AI stack serves both a browser frontend and host tools such as Codex, Claude Code, OpenCode, mi, Pi, or VS Code.

See [Run Coding Agents with Local LLMs](./8.3-Run-Coding-Agents-with-Local-LLMs.md) for the dedicated guide.

## Eject When You Want Plain Compose

Harbor does not lock you into the CLI. If you want to move a selected setup into a standalone Compose file, use `harbor eject`:

```bash
harbor eject searxng llamacpp > docker-compose.harbor.yml
```

`harbor eject` renders Harbor's selected Docker Compose configuration into a standalone file. This is useful when you used Harbor to find a working local AI stack but want to run or modify that setup outside Harbor afterward.

## When to Use Each Approach

Use manual Docker Compose when:

- You need a small, stable stack with one or two services
- You already have project-specific Compose conventions
- You want complete control over every YAML line
- You do not need Harbor's service catalog, config aliases, Launch adapters, App, or export workflow

Use Harbor when:

- You want a local AI stack with interchangeable frontends, backends, and satellites
- You expect to add services such as SearXNG, Speaches, ComfyUI, LiteLLM, Dify, n8n, Flowise, or LangFlow over time
- You want cross-service integrations maintained as part of the Harbor service catalog
- You want one CLI and optional app UI for starting, opening, configuring, inspecting, and exporting the stack
- You want local OpenAI-compatible backends to serve both web UIs and host coding tools

Harbor is best understood as a Compose-based local AI toolkit. It keeps Docker Compose underneath, adds a service catalog and integration layer on top, and leaves an escape hatch through `harbor cmd` and `harbor eject`.
