# Local LLM Stack with Docker Compose

Harbor is a local LLM stack manager for people who want a practical local AI stack without hand-wiring every container. It uses Docker Compose to run compatible frontends, inference backends, search tools, voice services, image tools, workflow services, and proxies as one coordinated setup.

This guide explains the shape of a Docker Compose LLM stack in Harbor, starting with the default Ollama and Open WebUI setup.

## Why a Local LLM Stack Becomes Multiple Services

A single model server is enough for some prompts, but most useful local AI workflows quickly need more pieces:

- A backend that loads and serves models, such as [Ollama](./2.2.1-Backend&colon-Ollama.md), [llama.cpp](./2.2.2-Backend&colon-llama.cpp.md), or [vLLM](./2.2.3-Backend&colon-vLLM.md)
- A frontend for chat and model interaction, such as [Open WebUI](./2.1.1-Frontend&colon-Open-WebUI.md)
- Optional web search for local web RAG, commonly through [SearXNG](./2.3.1-Satellite&colon-SearXNG.md)
- Optional speech, image, workflow, evaluation, and automation services
- Shared networking, ports, volumes, and service-specific configuration

Harbor treats these as a local LLM stack rather than a set of unrelated containers. The CLI selects the relevant Compose files, includes cross-service wiring when compatible services run together, and keeps common configuration in Harbor's profile and service override files.

## The Default Harbor Stack

The default Harbor setup starts Ollama and Open WebUI:

```bash
harbor up
```

That command brings up the default backend and frontend. Ollama serves local models, and Open WebUI provides the browser interface. You can then open the default frontend:

```bash
harbor open
```

This is the smallest useful Harbor local AI stack: one backend, one frontend, shared Docker networking, and Harbor-managed configuration.

## Docker Compose LLM Stack Layout

Harbor stores Compose files under `services/` and always includes the base `compose.yml`. Service handles decide which additional files are included.

For example, a default setup includes the base Compose file, the Ollama service, the Open WebUI service, and any matching integration files:

```bash
harbor cmd -h
```

The exact output depends on your system and selected services, but the pattern is the same:

```text
docker compose
 - compose.yml
 - services/compose.ollama.yml
 - services/compose.webui.yml
 - services/compose.x.ollama.webui.yml
```

The `compose.x.*.yml` files are cross-service integration files. They are applied when the required services are selected together. That is how Harbor can keep a service useful by itself while also adding extra configuration when another compatible service is present.

See [Harbor Compose Setup](./6.-Harbor-Compose-Setup.md) for the lower-level Compose matching rules.

## Adding Search, Voice, and Image Tools

Once the default stack is running, add services by naming their handles:

```bash
# Add local web search for Open WebUI and other compatible services
harbor up searxng

# Add OpenAI-compatible speech services used by compatible frontends
harbor up speaches

# Add local image generation workflows
harbor up comfyui
```

You can combine services in one command when you know the stack you want:

```bash
harbor up searxng speaches comfyui
```

Harbor will include the relevant service files and any compatible cross-service files for the selected handles. This keeps the operational model close to Docker Compose while avoiding repeated manual edits to Compose YAML for common local AI combinations.

## Swapping Backends

Ollama is Harbor's default backend because it is convenient for local model management. You can also run other backends when you need a different serving path:

```bash
harbor up llamacpp
harbor up vllm
```

Harbor supports multiple inference backends, including OpenAI-compatible servers, but the right backend depends on the model format, hardware, and workflow. Use the [Services catalog](./2.-Services.md#backends) to compare available backends.

## When Harbor Helps Most

Harbor is most useful when the local AI setup is more than one container:

- You want Open WebUI connected to a local backend without manual endpoint setup
- You want optional web search, voice, image generation, or workflow tools beside chat
- You want to switch between Ollama, llama.cpp, vLLM, and other backends
- You want to inspect or export the generated Compose command with `harbor cmd` or `harbor eject`
- You want a repeatable local AI stack controlled by one CLI and the same service handles

For a minimal first run, install Harbor, start the default stack, and open the UI:

```bash
harbor up
harbor open
```

From there, add the services that match the workflow you want instead of treating each service as a separate one-off Compose project.
