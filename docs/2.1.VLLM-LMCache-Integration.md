# vLLM LMCache Integration

LMCache provides distributed KV cache sharing for LLM inference servers like vLLM, enabling:
- **Faster time-to-first-token (TTFT)** when serving the same prompt from multiple models
- **Reduced memory usage** by reusing cached attention keys/values across requests
- **Improved throughput** by avoiding repeated computation for shared prefixes

## Configuration

### Environment Variables

These variables control the LMCache integration settings in `profiles/default.env`:

| Variable | Default | Description |
|----------|---------|-------------|
| `HARBOR_LMCACHE_L1_SIZE_GB` | `60` | CPU memory allocated for KV cache (GiB) |
| `HARBOR_LMCACHE_MAX_WORKERS` | `4` | Parallel worker threads |
| `HARBOR_LMCACHE_PORT` | `6555` | Server port on the internal network |

### vLLM Configuration

vLLM connects to LMCache via `--kv-transfer-config` flags expressed in **period notation**, appended to `HARBOR_VLLM_EXTRA_ARGS`:

```
--kv-transfer-config.kv_connector LMCacheMPConnector
--kv-transfer-config.kv_role kv_both
--kv-transfer-config.kv_connector_extra_config.lmcache.mp.host tcp://lmcache-server
--kv-transfer-config.kv_connector_extra_config.lmcache.mp.port 6555
```

Flags append to whatever other args are already set вҖ” the compose overlay does not override `HARBOR_VLLM_EXTRA_ARGS`, it only defines the `lmcache-server` sidecar.

## Usage

To run LMCache alongside vLLM:

```bash
# Start both services together
harbor up vllm
```

The integration automatically applies `compose.x.lmcache.vllm.yml` when both containers are running, providing shared memory access via `ipc: host` and GPU device reservations for CUDA IPC.

## Troubleshooting

**LMCache not found:** Verify the lmcache-server container is reachable by its service name:
```bash
harbor ps | grep -i lmcache
```

**Port conflicts:** LMCache uses port 6555 by default. Change `HARBOR_LMCACHE_PORT` and update vllm extra args accordingly.
