### [Cognee](https://github.com/topoteretes/cognee)

> Handle: `cognee`<br/>
> URL: [http://localhost:34761](http://localhost:34761)

Cognee is an open-source knowledge engine that transforms raw data into persistent, dynamic AI memory for agents. It combines vector search (LanceDB), graph databases (KuzuDB), and relational storage (SQLite) to make documents both searchable by meaning and connected by relationships. All databases are file-based by default -- no external database services required.

Harbor includes both the Cognee API server and the Cognee MCP server for IDE integration.

## Starting

```bash
# Build from source (first build takes several minutes)
harbor build cognee

# Start Cognee (API server + MCP server)
harbor up cognee
```

- API documentation (Swagger UI) is available at [http://localhost:34761/docs](http://localhost:34761/docs)
- MCP server (SSE transport) is available at [http://localhost:34762/sse](http://localhost:34762/sse)
- Authentication is disabled by default -- no registration or login needed.
- By default, Cognee uses Ollama for both LLM and embeddings -- start with Ollama for zero-config local operation:
  ```bash
  harbor up cognee ollama
  ```
- For other backends, see [Backend Integration](#backend-integration)

## Configuration

### Environment Variables

Following options can be set via [`harbor config`](./3.-Harbor-CLI-Reference.md#harbor-config):

```bash
# API port
HARBOR_COGNEE_HOST_PORT              34761

# Git reference for building from source
HARBOR_COGNEE_GIT_REF               https://github.com/topoteretes/cognee.git#main

# Workspace directory for persistent data
HARBOR_COGNEE_WORKSPACE              ./cognee

# LLM configuration (Ollama defaults)
HARBOR_COGNEE_LLM_API_KEY           ollama
HARBOR_COGNEE_LLM_MODEL             qwen3.5:9b
HARBOR_COGNEE_LLM_PROVIDER          ollama
HARBOR_COGNEE_LLM_ENDPOINT          http://ollama:11434/v1

# Embedding configuration (Ollama defaults)
HARBOR_COGNEE_EMBEDDING_PROVIDER    ollama
HARBOR_COGNEE_EMBEDDING_MODEL       nomic-embed-text:latest
HARBOR_COGNEE_EMBEDDING_ENDPOINT    http://ollama:11434/api/embed
HARBOR_COGNEE_EMBEDDING_DIMENSIONS  768

# HuggingFace tokenizer (used with Ollama embeddings)
HARBOR_COGNEE_HUGGINGFACE_TOKENIZER nomic-ai/nomic-embed-text-v1.5

# MCP server configuration
HARBOR_COGNEE_MCP_HOST_PORT          34762
HARBOR_COGNEE_MCP_IMAGE              cognee/cognee-mcp
HARBOR_COGNEE_MCP_VERSION            main
```

### Volumes

Cognee persists data in the following directories:
- `cognee/data/` - Data files (LanceDB vectors, KuzuDB graphs, SQLite database)
- `cognee/system/` - System files
- `cognee/cache/` - Cache files

## Backend Integration

Cognee defaults to Ollama for both LLM and embeddings. When you add another inference backend, its cross-file automatically overrides the LLM connection while keeping Ollama for embeddings.

### Ollama (default)

```bash
harbor up cognee ollama
```

### Other Backends

```bash
# vLLM for LLM, Ollama for embeddings
harbor up cognee vllm ollama

# llama.cpp for LLM, Ollama for embeddings
harbor up cognee llamacpp ollama

# Any OpenAI-compatible backend works
harbor up cognee tabbyapi ollama
harbor up cognee aphrodite ollama
harbor up cognee sglang ollama
```

Supported backends: Ollama, vLLM, llama.cpp, LiteLLM, TabbyAPI, Aphrodite, SGLang, mistral.rs, KTransformers, TGI, LMDeploy, AirLLM, KoboldCpp, Modular MAX, LocalAI, Nexa.

### Overriding Defaults

```bash
# Change the LLM model
harbor config set cognee.llm_model qwen2.5:7b

# Change the embedding model
harbor config set cognee.embedding_model mxbai-embed-large:latest
harbor config set cognee.embedding_dimensions 1024

# Switch to OpenAI entirely
harbor config set cognee.llm_provider openai
harbor config set cognee.llm_api_key sk-your-key-here
harbor config set cognee.llm_model openai/gpt-4o-mini
harbor config set cognee.llm_endpoint ""
harbor config set cognee.embedding_provider openai
harbor config set cognee.embedding_model openai/text-embedding-3-large
harbor config set cognee.embedding_endpoint ""
harbor config set cognee.embedding_dimensions 3072
```

## MCP Server

The Cognee MCP server starts automatically alongside the API server. It runs in API mode, connecting to the Cognee API over the internal Docker network.

### IDE Configuration

Add the following to your MCP client configuration:

**Claude Code** (`~/.claude.json`):
```json
{
  "mcpServers": {
    "cognee": {
      "type": "sse",
      "url": "http://localhost:34762/sse"
    }
  }
}
```

**Cursor** (`~/.cursor/mcp.json`):
```json
{
  "mcpServers": {
    "cognee": {
      "url": "http://localhost:34762/sse"
    }
  }
}
```

### Available MCP Tools

- `cognify` -- Build a knowledge graph from ingested data
- `search` -- Query the knowledge graph (supports GRAPH_COMPLETION, RAG_COMPLETION, CHUNKS, SUMMARIES, and more)
- `save_interaction` -- Log user-agent interactions
- `codify` -- Analyze a code repository and build a code graph (direct mode only)
- `delete` -- Remove specific data from a dataset
- `list_data` -- List datasets and their data items
- `prune` -- Reset all data for a fresh start (direct mode only)
- `cognify_status` -- Check the status of a cognify operation (direct mode only)
- `codify_status` -- Check the status of a codify operation (direct mode only)
- `cognee_add_developer_rules` -- Add rules to guide the knowledge graph creation process
- `get_developer_rules` -- Retrieve current developer rules (direct mode only)

> Tools marked "direct mode only" are unavailable via the MCP server since Harbor runs cognee-mcp in API mode. Use the [Cognee API](http://localhost:34761/docs) directly for those operations.

### MCPO Gateway

When running with [MCPO](./2.3.43-Satellite-mcpo.md), Cognee's MCP server is auto-registered in the gateway:

```bash
harbor up cognee mcpo
```

### Open WebUI

When running with [Open WebUI](./2.1.1-Frontend:-Open-WebUI.md), Cognee is automatically registered as an MCP tool server via a cross-file:

```bash
harbor up webui cognee
```

No manual configuration needed -- Harbor switches `cognee-mcp` from its default SSE mode to Streamable HTTP, keeps it connected to the Cognee API server, and pre-registers it in Open WebUI using the correct MCP connection type. Cognee tools (`search`, `cognify`, etc.) then appear in Open WebUI's tool picker while continuing to use the same local LLM/backend settings as the main Cognee service.

This integration is specific to `harbor up webui cognee`. Running `harbor up cognee` by itself still exposes the standalone SSE endpoint at [http://localhost:34762/sse](http://localhost:34762/sse) for IDE clients.

## Troubleshooting

### Check Logs

```bash
# API server logs
harbor logs cognee

# MCP server logs
harbor logs cognee-mcp
```

### Rebuild After Source Changes

```bash
harbor down cognee
harbor build cognee --no-cache
harbor up cognee
```

### Reset Data

```bash
harbor down cognee
rm -rf cognee/data cognee/system
harbor up cognee
```

### Search Hangs with Ollama

The default search type (GRAPH_COMPLETION) requires structured JSON output from the LLM. Smaller Ollama models may not produce valid JSON consistently, causing retries that appear as timeouts. Use CHUNKS search type instead, which retrieves matching text without LLM calls:

```bash
curl -X POST http://localhost:34761/api/v1/search \
  -H 'Content-Type: application/json' \
  -d '{"query": "your query", "search_type": "CHUNKS"}'
```

Alternatively, use a more capable model:
```bash
harbor config set cognee.llm_model qwen3:30b
```
