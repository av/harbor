### [MLX](https://github.com/raspoli/mlx-serve)

> Handle: `mlx`<br/>
> URL: [http://localhost:34930](http://localhost:34930)

MLX is Apple's machine-learning framework for Apple Silicon. Harbor integrates it through [mlx-serve](https://github.com/raspoli/mlx-serve): the MLX process runs natively on the macOS host for Metal access, while Harbor starts a proxy container so other services can use a normal Compose service URL.

This service is intended for Apple Silicon systems. On other platforms the proxy can still point at an externally managed compatible endpoint, but Harbor cannot provide MLX acceleration inside a Linux container.

#### Starting

Install `mlx-serve` on the host first:

```bash
uv tool install mlx-serve
```

Then start the Harbor service:

```bash
harbor up mlx
```

When `HARBOR_MLX_MANAGE_HOST=true`, Harbor renders `services/mlx/models.yaml` from the template, starts `mlx-serve` from `services/mlx/`, and starts the Harbor proxy.

Start a frontend against MLX:

```bash
harbor up webui mlx
```

Use it with host tools:

```bash
harbor launch --backend mlx --model mlx-qwen2.5-3b codex
```

#### Configuration

##### Environment Variables

Following options can be set via [`harbor config`](./3.-Harbor-CLI-Reference.md#harbor-config):

```bash
# Harbor proxy port
HARBOR_MLX_HOST_PORT              34930

# Proxy image
HARBOR_MLX_IMAGE                  caddy
HARBOR_MLX_VERSION                2-alpine

# Host workspace and upstream endpoint
HARBOR_MLX_WORKSPACE              ./services/mlx
HARBOR_MLX_UPSTREAM_URL           http://host.docker.internal:8095
HARBOR_MLX_RUNNER_PORT            8095
HARBOR_MLX_WORKER_PORT            8094

# Default model rendered into services/mlx/models.yaml
HARBOR_MLX_MODEL                  mlx-qwen2.5-3b
HARBOR_MLX_MODEL_TYPE             text
HARBOR_MLX_HF_PATH                mlx-community/Qwen2.5-3B-Instruct-4bit
HARBOR_MLX_CONTEXT_LENGTH         32768
HARBOR_MLX_MAX_KV_CACHE_SIZE      4096

# mlx-serve lifecycle settings
HARBOR_MLX_AUTO_UNLOAD            true
HARBOR_MLX_MAX_LOADED_MODELS      1
HARBOR_MLX_MANAGE_HOST            true
HARBOR_MLX_AUTO_PULL              true
HARBOR_MLX_API_KEY                sk-mlx
```

##### Volumes

Harbor mounts only `services/mlx/Caddyfile` into the proxy container. The host runner uses:

- `services/mlx/models.yaml.template` - Harbor-managed template
- `services/mlx/models.yaml` - generated mlx-serve config
- `services/mlx/cache/` - local cache directory
- `services/mlx/logs/` - local logs directory

#### Model Management

```bash
harbor models ls --source mlx
harbor models pull --source mlx mlx-qwen2.5-3b
harbor models rm --source mlx mlx-qwen2.5-3b
```

The equivalent source-subcommand form is also supported:

```bash
harbor models mlx pull mlx-qwen2.5-3b
```

`harbor mlx pull` delegates to the mlx-serve model management API. The default `harbor up mlx` flow also renders the configured model into `models.yaml` and attempts to pull it when `HARBOR_MLX_AUTO_PULL=true`.

#### API

Harbor exposes MLX at:

```bash
http://localhost:34930/v1
```

Containers use:

```bash
http://mlx:8080/v1
```

#### Troubleshooting

```bash
harbor mlx status
harbor mlx logs
harbor logs mlx
```

##### mlx-serve is not installed

Install it on the macOS host:

```bash
uv tool install mlx-serve
```

##### Proxy is unhealthy

Check the host runner:

```bash
harbor mlx status
curl http://localhost:8095/v1/models
```

If you manage mlx-serve yourself, set:

```bash
harbor config set mlx.manage.host false
harbor config set mlx.upstream.url http://host.docker.internal:8095
```
